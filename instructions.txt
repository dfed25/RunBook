Runbook: AI Onboarding Copilot
1. The One-Sentence Idea
Runbook is an AI onboarding copilot that turns scattered company knowledge into step-by-step
workflows new employees can actually complete.
Most AI workplace tools help people find information. Runbook helps new hires do the work.
Instead of a new employee asking:
“Where do I find the expense policy?”
Runbook answers:
“Here is the policy, here is the form, here are the steps, and I’ll track whether you finished it.”
That difference matters. We are not building another AI search bar. We are building an onboarding
execution system.
2. The Problem
Every company has the same problem: important knowledge is scattered everywhere.
It lives in:
Notion pages
Google Drive folders
Slack messages
onboarding docs
HR documents
random PDFs
email threads
people’s heads
When a new hire joins, they usually waste days or weeks asking questions like:
“Where is the design system?”
“How do I request access to GitHub?”
“Who do I ask about expenses?”
“What am I supposed to do on day one?”
“Where is the product roadmap?”
“How do we ship a feature here?”

This creates three big problems:
1. New hires are slow to become productive
Companies pay employees before they are fully useful. Bad onboarding burns time and money.
2. Existing employees get interrupted constantly
Senior employees, managers, and teammates answer the same questions over and over.
3. Company knowledge disappears
When employees leave, their knowledge often leaves with them.
3. Why Existing Tools Are Not Enough
There are already tools like Notion AI, Glean, Guru, Confluence, Microsoft Copilot, and Slack AI.
But most of them are still search-first.
They answer questions like:
“Where is the policy?”
But they usually do not turn that answer into a structured onboarding process.
The key difference:
Existing tools help people find information. Runbook helps people complete onboarding tasks.
That is our wedge.
We should not pitch Runbook as:
“AI that answers questions about company docs.”
That sounds too similar to existing products.
We should pitch it as:
“An AI onboarding copilot that turns company knowledge into executable workflows so new
hires become productive faster.”

4. Our Differentiation
Differentiator 1: Action-first, not search-first
Most tools stop at an answer.
Runbook continues into action.
Example:
User asks:
“How do I submit an expense?”
A normal AI search tool says:
“Here is the expense reimbursement policy.”
Runbook says:
“Here are the steps: 1. Open the reimbursement form. 2. Attach your receipt. 3. Select your
department. 4. Submit before Friday. I’ll mark this task complete after you finish.”
This makes Runbook feel like a copilot, not just a chatbot.
Differentiator 2: Built specifically for onboarding
Glean and Copilot are horizontal tools. They try to help everyone with everything.
Runbook focuses on one high-value workflow: onboarding.
That gives us a clearer user, clearer ROI, and a better demo.
Differentiator 3: Progress tracking
Managers do not just want employees to ask questions. They want to know:
Has the new hire completed setup?
What are they stuck on?
What docs are confusing?
Which tasks keep creating questions?
Runbook can show a manager dashboard with onboarding progress.

Differentiator 4: Turns messy docs into workflows
Companies often have documentation, but it is messy. Runbook can ingest scattered documents and
generate onboarding paths automatically.
Example:
From docs about GitHub, Slack, payroll, security, and product, Runbook creates:
Day 1 checklist
Engineering setup checklist
HR compliance checklist
First project checklist
Differentiator 5: Browser assistant layer
This is the Grammarly-style idea.
Runbook should not only live as a website. It should eventually follow the employee across work tools.
For example, when a new hire is on GitHub, Slack, Notion, Google Drive, Linear, Jira, or an internal tool,
Runbook can appear as a small floating assistant.
It can say:
“You are on the GitHub access page. Do you want help completing the engineering setup
task?”
This makes Runbook feel embedded into work, not trapped inside a dashboard.

6. Part One: Company Knowledge Ingestion
The company connects sources like:
Notion
Google Drive
Slack
PDFs
internal onboarding docs
company handbook
For a hackathon MVP, we do not need every real integration. We can start with:
uploaded PDFs / text files
manually pasted company docs
maybe Google Drive or Notion if time allows
The ingestion pipeline works like this:
User uploads or connects documents.
Backend extracts the text.
Text is split into chunks.
Each chunk is embedded using an embedding model.
Embeddings are stored in a vector database.
When a user asks a question, we retrieve the most relevant chunks.
The LLM answers using only those chunks and cites sources.
This is called RAG: retrieval-augmented generation.
For the demo, we can create a fake company workspace with documents like:
Employee Handbook
Engineering Setup Guide
Expense Policy
Security Policy
Product Overview
Slack Etiquette
First Week Checklist
This gives us complete control over the demo.

21. Recommended Tech Stack
Use the stack that lets us ship fastest.
Recommended Stack
Frontend: Next.js + React + Tailwind
UI: shadcn/ui or simple Tailwind components
Backend: Next.js API routes
Database: Supabase, Firebase, Convex, or even local JSON/mock data for MVP
AI: OpenAI, Anthropic, Gemini, or whatever API is easiest to access
Vector search: easiest option available
Supabase vector if someone knows it
Pinecone if already familiar
Chroma if using Python
Simple keyword search fallback if vector setup becomes painful
Important Note
Do not let vector database complexity kill the project.
If true embeddings take too long, use a simpler search approach:
Store documents as text chunks.
Use keyword matching or basic similarity.
Send the most relevant chunks into the LLM.
Make the answer cite the chunk titles.
The judges care more about the working product and pitch than whether the retrieval system is technically
perfect.

24. How to Use AI Tools During the Hackathon
Use AI constantly, but in the right way.
For Coding
Use ChatGPT, Claude, Cursor, GitHub Copilot, or similar tools to:
generate components
debug errors
write API routes
write prompts
create fake demo data
write tests
explain stack traces
But never blindly paste large chunks without understanding them.
Best AI Prompts for Coding
Debug prompt
Here is the error, here is the file, and here is what I expected to happen. Find
the bug and give me the smallest safe fix.
Component prompt
Build a clean React/Tailwind component for a new hire onboarding checklist. It
should support todo, in-progress, and complete states, and call onTaskUpdate
when a task is clicked.
API prompt
Write a Next.js API route that receives a user question, retrieves relevant
document chunks from an array, sends them to an LLM, and returns an answer with
source titles.

Refactor prompt
Clean this code without changing behavior. Keep it simple and hackathon-safe.

tighten the pitch
write judge Q&A answers

47. 3-Minute Pitch Script
Use this as the base.
Every company has the same hidden tax: onboarding. New hires spend days or weeks asking
where things are, how processes work, and who to talk to. That slows them down and
interrupts everyone around them.
The problem is that company knowledge is scattered across wiki pages, docs, Slack, PDFs,
and people’s heads. Existing AI tools help employees search that knowledge, but they are still
passive. They tell you where the answer is. They do not help you complete the work.
Runbook is an AI onboarding copilot that turns company knowledge into step-by-step
workflows. A new hire can ask, “What should I do today?” and Runbook gives them a
personalized checklist. They can ask, “How do I get GitHub access?” and Runbook answers
with the exact steps and source documents. Managers can see progress and identify where
new hires are getting stuck.
We also show where this goes next: Runbook can turn long docs into short onboarding
lessons and can follow employees across work tools like a browser assistant, guiding them
exactly when they need help.
Our wedge is onboarding because the pain is universal, measurable, and urgent. But the
bigger vision is becoming the action layer over company knowledge. Existing tools help
employees find information. Runbook helps employees complete work.
48. Judge Q&A Cheat Sheet
How is this different from Glean or Notion AI?
Glean and Notion AI are mainly knowledge search tools. Runbook is focused on onboarding
execution. We turn docs into tasks, guide new hires through workflows, track completion, and
show managers where people are stuck.
Why start with onboarding?
Onboarding is painful, universal, measurable, and high-value. Every growing company has
this problem, and reducing time-to-productivity has clear ROI.
60
Who pays?
HR, operations, founders, and engineering managers. The simplest model is per employee
per month.
What is the bigger vision?
Onboarding is the wedge. The bigger product is an action layer for internal company
workflows: IT setup, compliance, sales enablement, support playbooks, and training.
What if docs are outdated?
Runbook can surface documentation gaps by tracking repeated questions, missing answers,
and conflicting sources. Instead of hiding messy docs, it helps companies improve them.
Why would companies trust it?
Answers cite sources, and the product can be configured to only answer from approved
company knowledge. For risky actions, it guides users instead of making unauthorized
changes.
